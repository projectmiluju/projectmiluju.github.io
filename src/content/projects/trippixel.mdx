## 서비스 소개

TripPixel은 여행 계획의 처음부터 끝을 하나의 플랫폼에서 해결한다.
YouTube 영상 URL을 넣으면 AI가 영상에서 여행지를 추출해주고, 멤버들이 지도 위에서 장소를 고르고 투표해 일정을 만든다.

> "여행 유튜브 보다가 좋은 곳 발견했는데 어디 저장하지?" 에서 출발한 서비스.

**팀 구성**: PM 1 · 디자이너 2 · BE 3 · FE 3 (총 9명)
**내 역할**: FE 팀 3인 중 1인 — 후보지 선택, 장소 검색, 플랜 상세 페이지 담당

## 내가 담당한 기능

### 후보지 선택 페이지

지도 위에 후보지 핀을 표시하고, 선택 완료 시 플랜 상세로 이동하는 플로우 전체를 담당했다.

- 후보지 목록 API 연동 및 `CandidatePin` 오버레이 구현
- 선택된 후보지 기준으로 지도 `fitBounds` 자동 조정
- 후보지 선택 완료 → 플랜 상세 이동 시 TanStack Query 캐시 invalidation

```typescript
useEffect(() => {
  if (!mapRef.current || candidates.length === 0) return;
  const bounds = new google.maps.LatLngBounds();
  candidates.forEach((c) => bounds.extend({ lat: c.lat, lng: c.lng }));
  mapRef.current.fitBounds(bounds);
}, [candidates]);
```

### 장소 검색 탭 + 보관함

장소를 검색하고 보관함에 저장한 뒤 일정에 추가하는 탭 UI를 구현했다.
보관함이 비어있을 때 버튼 활성화 조건을 잘못 처리하는 버그를 수정하면서 상태 설계를 다듬었다.

### 플랜 상세 — Scrollspy

날짜별 일정을 사이드바에서 클릭하면 해당 섹션으로 이동하고, 스크롤 위치에 따라 현재 날짜를 하이라이팅하는 scrollspy를 구현했다.

```typescript
const observer = new IntersectionObserver(
  (entries) => {
    entries.forEach((entry) => {
      if (entry.isIntersecting) setActiveDay(entry.target.id);
    });
  },
  { threshold: 0.5 }
);
```

## 배운 점

**서버 상태 설계의 중요성**
여러 페이지에서 같은 데이터를 참조할 때, mutation 이후 어느 쿼리를 invalidate할지 명확히 설계하지 않으면 화면 간 데이터 불일치가 생긴다. TanStack Query의 queryKey 구조를 팀 차원에서 합의하고 시작하는 게 중요하다는 걸 몸으로 배웠다.

**Google Maps API는 레퍼런스가 적다**
카카오맵 대비 한국어 레퍼런스가 거의 없어서 공식 문서와 타입 정의를 직접 읽어야 했다. 덕분에 공식 문서를 빠르게 파악하는 방식이 늘었다.

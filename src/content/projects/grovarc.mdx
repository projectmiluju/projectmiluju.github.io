## 아이디어

매일 코드를 짜고, 버그를 잡고, 새 기술을 배운다. 그런데 한 달 뒤에 "요즘 뭘 배웠어?"라는 질문을 받으면 막막하다. 성장하고 있는 건 알겠는데 그게 말로 잘 안 나온다.

회고를 써야 한다는 건 알지만 백지 앞에 앉으면 손이 안 움직인다. 뭘 썼는지, 어디서 막혔는지 기억을 되살리는 것부터 30분이 걸린다.

**Grovarc의 답**: 매일 작업 로그를 짧게 기록하면 AI Agent가 패턴을 분석해서 회고 초안, 성장 리포트, 학습 로드맵을 자동으로 만들어준다. 기록은 사람이, 분석은 AI가.

---

## 구조

단일 모노레포에 4개의 앱이 공존한다.

```
grovarc/
├── apps/
│   ├── web/    # Next.js — 작업 로그 기록, 회고 대시보드
│   ├── api/    # Kotlin Spring Boot — REST API, 사용자/로그 관리
│   ├── ai/     # Python FastAPI — LangGraph Agent, LLaMA 3 추론
│   └── mcp/    # TypeScript MCP — Claude와 직접 연결되는 인터페이스
├── infra/
│   ├── terraform/   # AWS 인프라 IaC
│   └── k8s/         # Kubernetes 매니페스트
└── docs/
    ├── PRD.md
    └── adr/         # 기술 결정 기록
```

## 기술 선택 이유

**Kotlin (api)**
Java의 JVM 생태계와 Spring Boot 호환성을 유지하면서 코드를 간결하게 쓸 수 있다. 코루틴으로 AI 서버 응답 대기를 비동기 처리하기에도 자연스럽다. 익숙한 Spring 생태계를 유지하면서 새 언어를 프로덕션 수준으로 써보고 싶었다.

**Python + FastAPI (ai)**
LangGraph, LLaMA, RAG 생태계가 모두 Python에 있다. AI 서버만 Python으로 분리하고, 나머지는 Kotlin으로 유지하는 게 각 언어의 장점을 최대화하는 선택이었다.

**LangGraph**
단순 LLM 호출이 아니라 "패턴 분석 → 회고 초안 생성 → 로드맵 제안" 같은 복잡한 흐름이 필요하다. LangGraph는 이런 멀티 스텝 Agent 흐름을 그래프로 명확하게 표현할 수 있다.

**MCP SDK**
Claude Desktop과 직접 연결해서 사용자가 Claude에게 "지난 2주 동안 내가 뭘 배웠어?"라고 물으면 Grovarc 데이터를 기반으로 대답하게 만든다. 서비스를 열지 않아도 Claude 안에서 성장 데이터와 대화할 수 있다.

**Kubernetes + Terraform**
api, ai, mcp 각 앱을 독립 Deployment로 운영해 부하에 따라 개별 스케일 아웃이 가능하다. 인프라를 코드로 관리해 어떤 환경에서도 동일한 상태를 재현할 수 있다.

## 개발 방식

Cursor Agent + Claude를 활용한 AI-assisted 개발(바이브코딩)로 진행한다. 코드를 직접 짜는 속도보다 **무엇을 왜 만드는지를 먼저 명확히 하는 것**에 더 집중한다.

PRD 작성 → 아키텍처 문서 → ADR → 구현 순서를 지키고, 개발 과정은 [기술 블로그](/blog)에 기록한다.

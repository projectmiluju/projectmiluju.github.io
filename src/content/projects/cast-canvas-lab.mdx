## 문제 정의

리서치를 할 때 겪는 불편함:
- 문서, 이미지, 메모가 각각 다른 도구에 흩어져 있어 흐름 파악이 어려움
- Notion은 선형적이고, Figma는 문서 친화적이지 않음
- 아이디어 간 관계와 흐름을 공간적으로 표현할 수 있는 도구 부재

**목표**: PDF, 이미지, 노트를 하나의 무한 캔버스 위에 배치하고 연결해 리서치 흐름을 공간적으로 정리하는 워크스페이스

> Notion보다 더 공간적으로, Figma보다 더 문서 친화적으로.

## 역할과 범위

**역할**: 솔로 풀스택 개발자
**범위**:
- 워크스페이스 프론트엔드 (React + @xyflow/react)
- 메인 백엔드 API 서버 (Spring Boot)
- 실시간 협업 서버 (NestJS + Yjs)
- 퍼블릭 랜딩 사이트 (Next.js)
- 인프라 (AWS, GitHub Actions CI/CD)

## 핵심 구현

### 1. @xyflow/react 기반 캔버스

React Flow v12를 활용한 노드/엣지 기반 무한 캔버스. 각 노드는 PDF, 이미지, 노트 중 하나의 타입을 가짐:

```typescript
// 커스텀 노드 타입 정의
const nodeTypes = {
  document: DocumentNode,  // PDF 렌더링 노드
  image: ImageNode,        // 이미지 노드
  note: NoteNode,          // 텍스트 노트 노드
};

const WorkspaceCanvas = () => {
  const { nodes, edges, onNodesChange, onEdgesChange, onConnect } = useCanvasStore();

  return (
    <ReactFlow
      nodes={nodes}
      edges={edges}
      nodeTypes={nodeTypes}
      onNodesChange={onNodesChange}
      onEdgesChange={onEdgesChange}
      onConnect={onConnect}
      fitView
    >
      <Background />
      <Controls />
      <MiniMap />
    </ReactFlow>
  );
};
```

### 2. Yjs 기반 실시간 협업 (NestJS)

NestJS WebSocket 게이트웨이에서 Yjs 업데이트와 awareness를 브로드캐스트:

```typescript
// NestJS WebSocket 게이트웨이 (collab 서버)
@WebSocketGateway({ path: '/collab' })
export class CollabGateway {

  @SubscribeMessage('sync')
  handleSync(
    @MessageBody() update: Uint8Array,
    @ConnectedSocket() client: WebSocket,
  ) {
    // Yjs 업데이트를 같은 룸의 다른 클라이언트에 브로드캐스트
    this.broadcastToRoom(client.roomId, update, client);
  }

  @SubscribeMessage('awareness')
  handleAwareness(
    @MessageBody() awarenessUpdate: Uint8Array,
    @ConnectedSocket() client: WebSocket,
  ) {
    this.broadcastToRoom(client.roomId, awarenessUpdate, client);
  }
}
```

### 3. Spring Boot — 워크스페이스 API

캔버스 노드/엣지 메타데이터를 저장하고 조회하는 REST API:

```java
@RestController
@RequestMapping("/api/workspaces/{workspaceId}/canvas")
@RequiredArgsConstructor
public class CanvasController {

    private final CanvasService canvasService;

    @GetMapping("/nodes")
    public ResponseEntity<List<NodeDto>> getNodes(
        @PathVariable Long workspaceId,
        @AuthenticationPrincipal UserDetails user
    ) {
        return ResponseEntity.ok(canvasService.getNodes(workspaceId, user.getUsername()));
    }

    @PatchMapping("/nodes/{nodeId}")
    public ResponseEntity<NodeDto> updateNodePosition(
        @PathVariable Long workspaceId,
        @PathVariable Long nodeId,
        @RequestBody NodePositionRequest request
    ) {
        return ResponseEntity.ok(canvasService.updatePosition(nodeId, request));
    }
}
```

### 4. AWS 랜딩 사이트 배포 자동화

Next.js 랜딩 사이트를 S3 정적 호스팅 + CloudFront CDN으로 배포하고, GitHub Actions로 자동화:

```yaml
# .github/workflows/deploy-site.yml
- name: Build Next.js
  run: pnpm build

- name: Upload to S3
  run: aws s3 sync out/ s3://${{ secrets.S3_BUCKET }} --delete

- name: Invalidate CloudFront
  run: |
    aws cloudfront create-invalidation \
      --distribution-id ${{ secrets.CF_DISTRIBUTION_ID }} \
      --paths "/*"
```

## 트러블슈팅

### 이슈 1: 크로스 레포 계약 관리

**현상**: 5개 레포가 각자 독립적으로 개발되면서 FE↔BE API 계약, FE↔COLLAB WebSocket 프로토콜이 불일치할 위험

**해결**: `cast-canvas-lab-orchestrator` 레포를 별도로 운영하여 계약 문서(openapi.yaml, collab-protocol.md)를 단일 소스로 관리. 각 레포는 이 계약을 참조해 구현

---

### 이슈 2: PDF 렌더링 노드 성능

**현상**: PDF 노드가 여러 개 있을 때 캔버스 조작이 버벅거림

**원인**: 모든 PDF 페이지를 한 번에 렌더링하여 메모리/렌더링 부하

**해결**: react-pdf로 첫 페이지만 썸네일 렌더링, 노드 클릭 시 전체 페이지 로드

```typescript
// 썸네일 모드: 첫 페이지만 렌더링
<Document file={url}>
  <Page pageNumber={1} width={240} renderTextLayer={false} renderAnnotationLayer={false} />
</Document>
```

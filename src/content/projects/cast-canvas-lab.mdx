## 왜 만드는가

리서치를 할 때마다 같은 문제를 겪었다. 좋은 아티클은 노션에, 레퍼런스 이미지는 피그마에, 메모는 애플 노트에 흩어진다. 꺼내 볼 때마다 탭을 10개씩 열어야 하고, 아이디어 사이의 연결은 내 머릿속에만 존재한다.

노션은 선형적이고, 피그마는 문서를 다루기 불편하다. **PDF, 이미지, 노트를 하나의 캔버스 위에 올려두고 화살표로 연결하는 것** — 이게 내가 원하는 도구였다.

> Notion보다 더 공간적으로, Figma보다 더 문서 친화적으로.

## 구조

5개 레포가 각자의 역할을 가진다. 규모가 커지면 어느 레포에서 어느 작업을 해야 할지 혼란스러워지기 때문에, 처음부터 `orchestrator` 레포를 두고 API 계약과 작업 흐름을 한 곳에서 관리한다.

| 레포 | 역할 |
|------|------|
| [orchestrator](https://github.com/projectmiluju/cast-canvas-lab-orchestrator) | API 계약 (OpenAPI), 크로스 레포 태스크 관리 |
| [fe](https://github.com/projectmiluju/cast-canvas-lab-fe) | 워크스페이스 캔버스 앱 (React 19 + @xyflow/react) |
| [be](https://github.com/projectmiluju/cast-canvas-lab-be) | REST API 서버 (Spring Boot 3.5) |
| [collab](https://github.com/projectmiluju/cast-canvas-lab-collab) | 실시간 협업 서버 (NestJS + Yjs) |

## 기술 선택 이유

**@xyflow/react (React Flow v12)**
노드와 엣지로 요소를 연결하는 캔버스를 처음부터 구현하는 건 너무 복잡하다. React Flow는 드래그, 연결, 줌, 뷰포트를 모두 제공하면서 커스텀 노드를 React 컴포넌트로 자유롭게 만들 수 있다. PDF 노드, 이미지 노드, 노트 노드를 독립적인 컴포넌트로 구성할 수 있어서 선택했다.

**Yjs + NestJS (collab 서버)**
여러 사람이 같은 캔버스를 편집할 때 충돌을 해결하려면 CRDT가 필요하다. Yjs는 검증된 CRDT 구현체고, y-protocols로 WebSocket transport를 쉽게 구성할 수 있다. 별도 collab 서버로 분리한 이유는, 협업 연결 수가 늘어날 때 BE와 독립적으로 스케일 아웃하기 위해서다.

**Spring Boot (BE) + Flyway**
캔버스 메타데이터, 워크스페이스, 사용자 데이터는 관계형 구조가 맞다. Flyway로 스키마 변경을 코드로 추적하고, 로컬-프로덕션 간 DB 상태 불일치를 막는다.

## 현재까지 만든 것

**랜딩 사이트** (완료)
Next.js로 제품 소개 페이지를 만들고 AWS S3 + CloudFront + Route53으로 배포했다. `main` 브랜치 push 시 GitHub Actions로 자동 빌드·배포된다. → [castcanvaslab.com](https://castcanvaslab.com/)

**BE 기본 인프라** (완료)
Spring Boot 3.5 기반으로 공통 응답/에러 처리, 사용자 프로필 API, Flyway 마이그레이션 구조를 구축했다.

**collab 서버 설계** (완료)
NestJS WebSocket 게이트웨이, Yjs 업데이트 브로드캐스트, awareness 전달 구조를 설계했다.
